# Widening `id` to `BIGINT`

Before [#671](https://github.com/janbjorge/pgqueuer/issues/671) the `queue`,
`statistics`, and `schedules` tables used `id SERIAL PRIMARY KEY` — a 32-bit
integer capped at 2,147,483,647. The backing sequence only climbs and never
reuses values, so a long-lived, high-throughput deployment could exhaust it and
then fail **every** `enqueue` with an integer-out-of-range / sequence-limit
error.

Fresh installs now create these ids as `BIGINT GENERATED BY DEFAULT AS IDENTITY`
(matching `queue_log`, which was always `BIGINT`). Existing installs are
migrated by `pgq upgrade`.

## What `pgq upgrade` does

For each affected table it widens both the column and its legacy sequence:

```sql
ALTER TABLE pgqueuer            ALTER COLUMN id TYPE BIGINT;
ALTER TABLE pgqueuer_statistics ALTER COLUMN id TYPE BIGINT;
ALTER TABLE pgqueuer_schedules  ALTER COLUMN id TYPE BIGINT;
```

Widening the column alone is not enough: a `SERIAL` column's sequence was
created `AS integer` and still caps at 2^31-1, so `pgq upgrade` also widens each
backing sequence:

```sql
ALTER SEQUENCE pgqueuer_id_seq AS BIGINT;  -- and the statistics / schedules sequences
```

Both steps are wrapped in `DO` blocks guarded on the current type, so the
migration is **idempotent** — it is a no-op once everything is already `BIGINT`
and is safe to re-run.

## The migration takes an `ACCESS EXCLUSIVE` lock

!!! warning
    `ALTER COLUMN ... TYPE BIGINT` rewrites the whole table and rebuilds its
    indexes. Postgres holds an `ACCESS EXCLUSIVE` lock for the entire rewrite,
    which blocks **everything** on the table — enqueues, dequeues, and even
    plain `SELECT`s. The rewrite time scales with row count, so on a large or
    bloated `queue` table this can stall the queue for seconds to minutes.
    While the statement *waits* to acquire the lock, new queries queue up behind
    it, so a single long-running transaction can freeze the queue for the whole
    wait.

For a normally-drained queue (rows are deleted as jobs complete) the tables are
small and the rewrite is effectively instant. The hazard is real only for large,
bloated, or high-retention tables.

**Run `pgq upgrade` during a maintenance window or low-traffic period.**

## Skipping the widen

If you cannot take the lock during a routine upgrade, skip just the widen and
apply every other migration:

```bash
pgq upgrade --no-widen-id
```

This is equivalent to setting `widen_id=False` on `DBSettings`. Perform the
widen out-of-band (see below), then a later `pgq upgrade` (with the widen
enabled) is a no-op for the id columns because the guards detect `BIGINT`.

## Zero-downtime widen for large tables

Instead of rewriting the column in place, add a new `BIGINT` column, backfill it
online, and swap it in as the primary key using the technique from
[postgres.ai howto #0033](https://github.com/postgres-ai/postgres-howtos/blob/main/0033_how_to_redefine_a_PK_without_downtime.md).
The only exclusive lock is a near-instant catalog swap that reuses an index
built `CONCURRENTLY` and skips the not-null scan.

!!! danger "Rehearse first"
    This is a hand-applied runbook. Adapt the table/index names to your
    `PGQUEUER_PREFIX`, rehearse on a production-sized clone, and set a short
    `lock_timeout` (with retries) on every locking statement so a waiter cannot
    freeze the queue.

Default object names are assumed (`pgqueuer`, sequence `pgqueuer_id_seq`).

### 1. Add the new column (instant, no rewrite)

```sql
ALTER TABLE pgqueuer ADD COLUMN id_new BIGINT;
```

### 2. Mirror new rows before backfilling

```sql
CREATE FUNCTION pgqueuer_sync_id_new() RETURNS trigger AS $$
BEGIN
    NEW.id_new := NEW.id;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER pgqueuer_sync_id_new
    BEFORE INSERT OR UPDATE ON pgqueuer
    FOR EACH ROW EXECUTE FUNCTION pgqueuer_sync_id_new();
```

### 3. Backfill existing rows in batches

Repeat until zero rows are updated:

```sql
WITH batch AS (
    SELECT id FROM pgqueuer WHERE id_new IS NULL ORDER BY id LIMIT 10000
)
UPDATE pgqueuer t SET id_new = t.id FROM batch WHERE t.id = batch.id;
```

### 4. Prove not-null without a full exclusive scan

```sql
ALTER TABLE pgqueuer
    ADD CONSTRAINT pgqueuer_id_new_not_null CHECK (id_new IS NOT NULL) NOT VALID;
ALTER TABLE pgqueuer VALIDATE CONSTRAINT pgqueuer_id_new_not_null;
```

`VALIDATE` scans under `SHARE UPDATE EXCLUSIVE` — reads and writes continue. On
PG12+ the later `ADD PRIMARY KEY` reuses this validated constraint and skips its
own not-null scan.

### 5. Build the replacement indexes concurrently

Build a unique index for the new PK plus a copy of every index that references
`id`. `CONCURRENTLY` takes no long lock; the temporary names are renamed after
the swap.

```sql
CREATE UNIQUE INDEX CONCURRENTLY pgqueuer_pkey_new ON pgqueuer (id_new);
CREATE INDEX CONCURRENTLY pgqueuer_priority_id_id1_idx_new
    ON pgqueuer (priority ASC, id_new DESC) INCLUDE (id_new) WHERE status = 'queued';
CREATE INDEX CONCURRENTLY pgqueuer_updated_id_id1_idx_new
    ON pgqueuer (updated ASC, id_new DESC) INCLUDE (id_new) WHERE status = 'picked';
CREATE INDEX CONCURRENTLY pgqueuer_heartbeat_id_id1_idx_new
    ON pgqueuer (heartbeat ASC, id_new DESC) INCLUDE (id_new) WHERE status = 'picked';
CREATE INDEX CONCURRENTLY pgqueuer_ep_prio_id_idx_new
    ON pgqueuer (entrypoint, priority DESC, id_new ASC) WHERE status = 'queued';
```

A failed `CONCURRENTLY` build leaves an `INVALID` index — drop it and retry
before continuing.

### 6. The swap (short `ACCESS EXCLUSIVE`, no rewrite)

One transaction does the cutover: no table rewrite, no index build, no not-null
scan — only catalog changes.

```sql
SET lock_timeout = '2s';  -- retry the whole transaction on failure

BEGIN;

-- Detach id generation from the old column.
-- Legacy SERIAL installs default to nextval('pgqueuer_id_seq'):
ALTER TABLE pgqueuer ALTER COLUMN id DROP DEFAULT;
-- Identity installs instead: ALTER TABLE pgqueuer ALTER COLUMN id DROP IDENTITY IF EXISTS;

-- Promote the new unique index to the primary key (reuses the index, skips scan).
ALTER TABLE pgqueuer DROP CONSTRAINT pgqueuer_pkey;
ALTER TABLE pgqueuer ADD CONSTRAINT pgqueuer_pkey PRIMARY KEY USING INDEX pgqueuer_pkey_new;

-- Drop the old column; its dependent indexes and the sync trigger go with it.
DROP TRIGGER pgqueuer_sync_id_new ON pgqueuer;
DROP FUNCTION pgqueuer_sync_id_new();
ALTER TABLE pgqueuer DROP COLUMN id;

-- Rename the new column and indexes to the canonical names.
ALTER TABLE pgqueuer RENAME COLUMN id_new TO id;
ALTER INDEX pgqueuer_priority_id_id1_idx_new  RENAME TO pgqueuer_priority_id_id1_idx;
ALTER INDEX pgqueuer_updated_id_id1_idx_new   RENAME TO pgqueuer_updated_id_id1_idx;
ALTER INDEX pgqueuer_heartbeat_id_id1_idx_new RENAME TO pgqueuer_heartbeat_id_id1_idx;
ALTER INDEX pgqueuer_ep_prio_id_idx_new       RENAME TO pgqueuer_ep_prio_id_idx;

-- Re-attach id generation, reusing the (BIGINT-capable) sequence.
ALTER SEQUENCE pgqueuer_id_seq OWNED BY pgqueuer.id;
ALTER TABLE pgqueuer ALTER COLUMN id SET DEFAULT nextval('pgqueuer_id_seq');

ALTER TABLE pgqueuer DROP CONSTRAINT pgqueuer_id_new_not_null;

COMMIT;
```

### 7. Verify

```sql
SELECT data_type FROM information_schema.columns
WHERE table_name = 'pgqueuer' AND column_name = 'id';   -- expect: bigint
SELECT indexrelid::regclass, indisvalid
FROM pg_index WHERE indrelid = 'pgqueuer'::regclass;     -- all VALID
```

Then enqueue and dequeue a test job.

## Notes

- Apply the same pattern to `pgqueuer_statistics` / `pgqueuer_schedules` only if
  those tables are also large; they almost never are.
- `pgqueuer_log.job_id` is a plain `BIGINT` column, **not** a foreign key, so
  nothing references `pgqueuer.id` by constraint — no FK rewrites are needed.
- Index names must match exactly; a typo in the `RENAME` steps leaves the schema
  inconsistent with what `pgq upgrade` expects on future runs.
