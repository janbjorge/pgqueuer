# DRAFT ŌĆö Zero-downtime `id` widening (`int4` ŌåÆ `BIGINT`) for large tables

!!! danger "Draft / review before use"
    This is a hand-applied runbook for operators whose `pgqueuer` table is too
    large to absorb the plain `pgq upgrade` rewrite (see
    [Upgrading](upgrading.md#id-columns-widened-to-bigint-671)). It is **not**
    executed by `pgq upgrade`. Read every step, adapt the table/index names to
    your `--prefix`, and rehearse on a clone before running in production.

## Why a special procedure

`pgq upgrade` runs `ALTER TABLE pgqueuer ALTER COLUMN id TYPE BIGINT`, which
rewrites the whole table under an `ACCESS EXCLUSIVE` lock. On a small queue
table that is microseconds; on a large or bloated one it blocks every enqueue,
dequeue, and `SELECT` for the duration of the rewrite.

The zero-downtime approach never widens the column in place. Instead it adds a
new `BIGINT` column, backfills it online, and then swaps it in as the primary
key using the technique from
[postgres.ai howto #0033](https://github.com/postgres-ai/postgres-howtos/blob/main/0033_how_to_redefine_a_PK_without_downtime.md).
The only exclusive lock is the final catalog swap, which is near-instant because
it reuses an index built `CONCURRENTLY` and skips the not-null scan.

Default object names are assumed (`pgqueuer`, sequence `pgqueuer_id_seq`). Adjust
for your `PGQUEUER_PREFIX`.

## Lock-timeout guard

Every statement that takes a lock should fail fast rather than queue behind a
long transaction (a waiting `ACCESS EXCLUSIVE` request blocks new queries behind
it). Set a short timeout per session and retry on `lock_not_available`:

```sql
SET lock_timeout = '2s';
```

## Step 1 ŌĆö Add the new column (instant, no rewrite)

A nullable column with no default is a catalog-only change:

```sql
ALTER TABLE pgqueuer ADD COLUMN id_new BIGINT;
```

## Step 2 ŌĆö Keep new rows in sync

Add a trigger that mirrors `id` into `id_new` for every insert/update. Create it
**before** backfilling so no row is missed:

```sql
CREATE FUNCTION pgqueuer_sync_id_new() RETURNS trigger AS $$
BEGIN
    NEW.id_new := NEW.id;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER pgqueuer_sync_id_new
    BEFORE INSERT OR UPDATE ON pgqueuer
    FOR EACH ROW EXECUTE FUNCTION pgqueuer_sync_id_new();
```

## Step 3 ŌĆö Backfill existing rows in batches

Copy in bounded chunks so no single statement locks many rows for long. Repeat
until zero rows are updated:

```sql
WITH batch AS (
    SELECT id FROM pgqueuer
    WHERE id_new IS NULL
    ORDER BY id
    LIMIT 10000
)
UPDATE pgqueuer t
SET id_new = t.id
FROM batch
WHERE t.id = batch.id;
```

(In practice the queue table is mostly drained continuously, so the backlog of
`id_new IS NULL` rows is usually small.)

## Step 4 ŌĆö Prove not-null without a full exclusive scan

```sql
ALTER TABLE pgqueuer
    ADD CONSTRAINT pgqueuer_id_new_not_null CHECK (id_new IS NOT NULL) NOT VALID;

ALTER TABLE pgqueuer VALIDATE CONSTRAINT pgqueuer_id_new_not_null;
```

`VALIDATE` scans the table under a `SHARE UPDATE EXCLUSIVE` lock ŌĆö concurrent
reads and writes continue. On PG12+ the later `ADD PRIMARY KEY` reuses this
validated constraint and skips its own not-null scan.

## Step 5 ŌĆö Build the replacement indexes concurrently

Build a unique index for the new PK, plus a new copy of every index that
references `id`. `CONCURRENTLY` takes no long lock. Use temporary names; they are
renamed to the canonical names after the swap.

```sql
CREATE UNIQUE INDEX CONCURRENTLY pgqueuer_pkey_new
    ON pgqueuer (id_new);

CREATE INDEX CONCURRENTLY pgqueuer_priority_id_id1_idx_new
    ON pgqueuer (priority ASC, id_new DESC) INCLUDE (id_new) WHERE status = 'queued';

CREATE INDEX CONCURRENTLY pgqueuer_updated_id_id1_idx_new
    ON pgqueuer (updated ASC, id_new DESC) INCLUDE (id_new) WHERE status = 'picked';

CREATE INDEX CONCURRENTLY pgqueuer_heartbeat_id_id1_idx_new
    ON pgqueuer (heartbeat ASC, id_new DESC) INCLUDE (id_new) WHERE status = 'picked';

CREATE INDEX CONCURRENTLY pgqueuer_ep_prio_id_idx_new
    ON pgqueuer (entrypoint, priority DESC, id_new ASC) WHERE status = 'queued';
```

If any `CONCURRENTLY` build fails it leaves an `INVALID` index ŌĆö drop it and
retry before continuing.

## Step 6 ŌĆö The swap (short `ACCESS EXCLUSIVE`, no rewrite)

One transaction does the cutover. It is fast: no table rewrite, no index build,
no not-null scan ŌĆö only catalog changes.

```sql
SET lock_timeout = '2s';  -- retry the whole transaction on failure

BEGIN;

-- Detach the identity/sequence from the old column.
-- Legacy SERIAL installs: the default is nextval('pgqueuer_id_seq').
-- Post-fix installs: the column is GENERATED BY DEFAULT AS IDENTITY.
ALTER TABLE pgqueuer ALTER COLUMN id DROP DEFAULT;            -- SERIAL installs
-- ALTER TABLE pgqueuer ALTER COLUMN id DROP IDENTITY IF EXISTS;  -- identity installs

-- Promote the new unique index to the primary key (reuses the index, skips scan).
ALTER TABLE pgqueuer DROP CONSTRAINT pgqueuer_pkey;
ALTER TABLE pgqueuer ADD CONSTRAINT pgqueuer_pkey PRIMARY KEY USING INDEX pgqueuer_pkey_new;

-- Drop the old column. Its dependent indexes (old PK + the four id indexes) and
-- the sync trigger are dropped with it.
DROP TRIGGER pgqueuer_sync_id_new ON pgqueuer;
DROP FUNCTION pgqueuer_sync_id_new();
ALTER TABLE pgqueuer DROP COLUMN id;

-- Rename the new column and indexes to the canonical names.
ALTER TABLE pgqueuer RENAME COLUMN id_new TO id;
ALTER INDEX pgqueuer_priority_id_id1_idx_new  RENAME TO pgqueuer_priority_id_id1_idx;
ALTER INDEX pgqueuer_updated_id_id1_idx_new   RENAME TO pgqueuer_updated_id_id1_idx;
ALTER INDEX pgqueuer_heartbeat_id_id1_idx_new RENAME TO pgqueuer_heartbeat_id_id1_idx;
ALTER INDEX pgqueuer_ep_prio_id_idx_new       RENAME TO pgqueuer_ep_prio_id_idx;

-- Re-attach id generation. Reuse the existing sequence so values keep climbing
-- from where they were; it is already BIGINT-capable.
ALTER SEQUENCE pgqueuer_id_seq OWNED BY pgqueuer.id;       -- SERIAL installs
ALTER TABLE pgqueuer ALTER COLUMN id SET DEFAULT nextval('pgqueuer_id_seq');  -- SERIAL installs
-- For a clean match with fresh installs, convert to identity instead:
-- ALTER TABLE pgqueuer ALTER COLUMN id ADD GENERATED BY DEFAULT AS IDENTITY;
-- SELECT setval(pg_get_serial_sequence('pgqueuer', 'id'), (SELECT max(id) FROM pgqueuer));

-- Clean up the helper constraint.
ALTER TABLE pgqueuer DROP CONSTRAINT pgqueuer_id_new_not_null;

COMMIT;
```

## Step 7 ŌĆö Verify

```sql
\d pgqueuer
SELECT data_type FROM information_schema.columns
WHERE table_name = 'pgqueuer' AND column_name = 'id';   -- expect: bigint
```

Confirm the four secondary indexes and the primary key are present and `VALID`
(`SELECT indexrelid::regclass, indisvalid FROM pg_index WHERE indrelid =
'pgqueuer'::regclass;`), then enqueue/dequeue a test job.

## Notes and caveats

- Apply the same pattern to `pgqueuer_statistics` / `pgqueuer_schedules` only if
  those tables are also large; they almost never are.
- `pgqueuer_log.job_id` is a plain `BIGINT` column, **not** a foreign key, so
  nothing references `pgqueuer.id` by constraint ŌĆö no FK rewrites are needed.
- The swap transaction still takes a brief `ACCESS EXCLUSIVE` lock. With
  `lock_timeout` + retry it is sub-second and does not rewrite data.
- Rehearse on a production-sized clone. Index names must match exactly; a typo
  in the `RENAME` steps leaves the schema inconsistent with what `pgq upgrade`
  expects on future runs.
